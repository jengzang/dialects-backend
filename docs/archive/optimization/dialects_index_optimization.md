# dialects 数据库索引优化报告

## 问题总结

### 发现的问题

1. **错误的索引策略**：创建了 `(簡稱, 聲母/韻母/聲調)` 索引，误导 SQLite 查询优化器
2. **性能严重下降**：`/compare/ZhongGu` API 受影响，性能下降 10 倍
3. **冗余索引**：多个索引功能重叠，浪费存储空间

### 影响分析

| API | 受影响情况 | 原因 |
|-----|-----------|------|
| `/api/ZhongGu` | ✅ 无影响 | 使用最优索引 `(簡稱, 漢字)` |
| `/compare/ZhongGu` | ❌ 严重影响 | 被误导使用 `(簡稱, 聲母)` 索引 |

**性能对比：**
- 使用特征索引：扫描 103,306 行，耗时 43ms
- 使用主索引：扫描 397 行，耗时 4ms
- **差异：10 倍性能下降，260 倍扫描量增加**

## 为什么特征索引有害？

### 问题根源

**查询条件：**
```sql
WHERE 簡稱 IN (20个) AND 漢字 IN (500个)
GROUP BY 簡稱, 聲母
```

**SQLite 的错误选择：**
- 看到 `GROUP BY 簡稱, 聲母`
- 认为使用 `(簡稱, 聲母)` 索引可以避免排序
- **忽略了 WHERE 条件的选择性**

**实际执行：**
1. 使用 `(簡稱, 聲母)` 索引扫描所有 `(地点, 声母)` 组合
2. 扫描 103,306 行（20个地点 × 平均5000个声母组合）
3. 过滤 `漢字 IN (500个)`，只剩 397 行
4. 创建临时 B-tree 做 `COUNT(DISTINCT 漢字)`

**正确做法：**
1. 使用 `(簡稱, 漢字)` 索引直接定位 397 行
2. 在内存中做 `GROUP BY` 和聚合
3. 避免扫描 10 万+行

### 索引选择性分析

| 索引 | 匹配行数 | 选择性 | 效率 |
|------|---------|--------|------|
| `(簡稱, 漢字)` | 397 | 高 | ✅ 优秀 |
| `(簡稱, 聲母)` | 103,306 | 低 | ❌ 糟糕 |

## 最佳实践建议

### 索引设计原则

1. **匹配 WHERE 条件**：索引列应该匹配查询的 WHERE 条件
2. **考虑选择性**：优先索引选择性高的列组合
3. **避免过度索引**：每个索引都有维护成本
4. **测试查询计划**：使用 `EXPLAIN QUERY PLAN` 验证索引使用

### 查询优化原则

1. **使用最优索引**：必要时用 `INDEXED BY` 强制指定
2. **应用层聚合**：简单的聚合在应用层做可能更快
3. **避免 COUNT(DISTINCT)**：需要创建临时 B-tree，开销大
4. **批量查询**：一次查询所有数据，避免多次查询
