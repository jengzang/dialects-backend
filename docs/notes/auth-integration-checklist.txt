Interpretation
--------------

1. Resend path is disabled because both RESEND_API_KEY and RESEND_FROM_EMAIL are required.

Remaining real-world acceptance tasks
-------------------------------------
A. Email delivery smoke test
1. Configure one real mail path.
   Option A: Resend
   - export RESEND_API_KEY=...
   - export RESEND_FROM_EMAIL=verified-sender@example.com
   Option B: SMTP
   - export SMTP_HOST=...
   - export SMTP_PORT=587
   - export SMTP_USERNAME=...
   - export SMTP_PASSWORD=...
   - export SMTP_FROM=...

2. Optionally configure frontend redirect target:
   - export FRONTEND_VERIFY_EMAIL_URL=https://<frontend-host>/verify-email

3. Start the backend and trigger:
   POST /api/auth/register-email
   body: {"email":"<real inbox>"}

4. Acceptance:
   - inbox receives email
   - email body contains a working verification link
   - GET /api/auth/verify-email-registration?token=... returns ready_to_complete=true
   - POST /api/auth/complete-email-registration returns access_token / refresh_token / session_id

B. Google real-provider acceptance
1. Confirm Google console redirect URI exactly matches backend callback endpoint.
2. Trigger /api/auth/google/auth/start and complete the browser login.
3. Acceptance:
   - callback returns login/register payload with session_id
   - bind flow returns provider=google + provider_subject
   - rebind flow replaces the existing google identity for the same user

C. WeChat Web real-provider acceptance
1. Confirm WeChat app credentials, callback domain, and redirect URI.
2. Trigger /api/auth/wechat/web/auth/start and complete QR/auth flow.
3. Acceptance:
   - callback returns login/register payload with session_id
   - bind flow returns provider=wechat + provider_subject
   - rebind flow replaces the existing wechat identity for the same user

D. WeChat Mini acceptance
1. Confirm Mini app credentials and the frontend/app-side code-exchange choreography expected by the backend mini endpoints.
2. Exercise:
   - /api/auth/wechat/mini/auth
   - /api/auth/wechat/mini/register
   - /api/auth/wechat/mini/bind
   - /api/auth/wechat/mini/rebind
3. Acceptance:
   - auth returns login/register guidance for a real mini identity
   - register closes the user/session creation path correctly
   - bind/rebind behave consistently with replacement semantics

Known remaining risk
--------------------
- Google and WeChat Web are locally closed at the backend-contract level, but not yet proven against live provider credentials.
- WeChat Mini routes are now proven live in the current backend, but acceptance coverage and dedicated tests still need expansion.
- Email registration flow is locally closed, but real delivery is not yet available in the current environment because sender configuration is incomplete.
- Admin / analytics adaptation and login-method analytics/logging still appear to be partially implemented.
